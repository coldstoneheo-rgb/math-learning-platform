# 학생 가입 및 사용자 역할/권한 체계 개선 요청서

> 대상 프로젝트: 수학 학습 분석 플랫폼  
> 대상 브랜치: `codex/enhancement-docs`  
> 문서 버전: v2.0  
> 문서 목적: 코드 수정 전, 학생 가입 불가 문제와 상용화 이후의 사용자 역할·권한·서비스 범위·대시보드·학생용 말투 정책을 명확히 정의한다.  
> 작성 원칙: 본 문서는 **개선 요청서**이며, 코드 구현이 아닌 요구사항·권한 설계·수용 기준·우선순위 문서이다.

---

## 1. 요청 배경

현재 플랫폼의 실제 사용자는 다음 4개 그룹이다.

1. **나 / Super Admin / 시스템 관리자 / 서비스 운영자**
2. **교사 / 고객 / 개인 과외 교사 / 학원·강사 조직**
3. **학부모**
4. **학생**

테스트 중 학생이 직접 가입하려고 했을 때 회원가입 화면의 “사용자 유형” 선택 항목에 **학생**이 없어 가입 자체가 불가능한 문제가 확인되었다.

플랫폼에는 이미 학생 전용 대시보드, 학생 리포트 조회, 내 풀이 분석, 학습 계획, 성장 예측, 성취 배지, 동기부여 메시지 같은 학생 중심 기능이 존재한다. 하지만 학생 계정 생성·학생 정보 연결·학생 로그인 라우팅이 완결되지 않아, 학생 기능이 실제 서비스 흐름으로 연결되지 못하고 있다.

추가로, 상용화 이후에는 **교사도 고객**이다. 현재 주 타깃은 개인 과외 교사이지만, 향후 학원·소규모 교육 기관·그룹 수업 강사·교육 컨설턴트까지 “교사” 범주에 포함될 수 있다. 따라서 시스템 전체를 관장하는 **Super Admin인 나**와, 서비스를 구매·사용하는 **고객으로서의 교사** 사이의 권한 차이를 명확히 분리해야 한다.

## 2. 현재 확인된 문제 요약

### 2.1 학생 역할은 타입과 백엔드 일부에는 존재하지만, 가입 UI에는 없다

현재 타입 정의상 `UserRole`에는 `teacher`, `parent`, `student`가 포함되어 있고, `/api/auth/create-user`는 `teacher`, `parent`, `student`를 허용한다.

그러나 회원가입 페이지의 form state는 `teacher | parent`만 허용하고, select option도 `선생님`, `학부모`만 제공한다. 따라서 학생은 UI에서 가입할 수 없다.

### 2.2 학생 로그인 후 라우팅이 `/student`로 가지 않는다

로그인 페이지는 `teacher`이면 `/admin`, `parent`이면 `/parent`로 보내지만, 그 외 역할은 `/`로 이동시킨다. 따라서 설령 학생 계정이 생성되어도 학생은 로그인 후 학생 대시보드로 자동 진입하지 못한다.

### 2.3 학생 대시보드는 존재하지만 `students.user_id` 연결을 전제로 한다

학생 대시보드는 `users.role === 'student'`인지 확인한 뒤 `students.user_id = auth.user.id` 조건으로 학생 정보를 조회한다. 학생 리포트 상세 페이지도 동일하게 `students.user_id`를 사용해 본인 학생 레코드를 찾는다.

하지만 현재 `Student` 타입에는 `user_id`가 명시되어 있지 않으며, 성취 배지 마이그레이션 주석에도 “현재 students 테이블에 user_id 컬럼이 없으므로 학생 로그인 기능 구현 시 별도 마이그레이션 필요”라는 취지의 주석이 있다. 즉, 학생 기능의 핵심 연결키가 요구되지만 스키마·타입·RLS가 일관되게 준비되어 있지 않다.

### 2.4 Super Admin과 Teacher 고객 역할이 구분되지 않는다

요구사항상 사용자는 “나(Admin), 교사, 학부모, 학생”으로 구분되어야 한다. 그러나 현재 타입 정의의 `UserRole`은 `teacher | parent | student`이며, 현재 `/admin` 화면은 `teacher` 역할만 허용한다.

즉 현재 구현에서는 “교사”가 사실상 관리자 화면을 사용한다. 베타 이후 고객 교사를 받기 시작하면, 교사가 서비스 전체 설정·전체 고객·전체 과금·전체 데이터에 접근해서는 안 된다.

따라서 다음 구분이 필요하다.

| 구분 | 의미 | 핵심 차이 |
|---|---|---|
| **Super Admin / 나** | 플랫폼 전체 운영자 | 모든 테넌트, 과금, 시스템 설정, 기능 플래그, 장애 대응, 사용자 지원 권한 |
| **Teacher / 고객 교사** | 서비스를 구매·사용하는 교육자 | 본인 또는 본인 조직의 학생·학부모·리포트·수업 데이터만 관리 |

### 2.5 “교사”의 범주가 확장될 예정이다

현재 주 타깃은 개인 과외 교사이지만, 향후 아래 사용자도 교사 고객 범주에 포함될 수 있다.

- 1:1 개인 과외 교사
- 소규모 공부방 운영자
- 학원 원장 또는 학원 소속 강사
- 그룹 수업 강사
- 교육 컨설턴트
- 시험 대비·입시 수학 전문 강사

따라서 단순히 `teacher` 한 명이 전체 학생을 관리하는 구조가 아니라, 장기적으로는 **조직/테넌트 기반 구조**가 필요하다.

### 2.6 학생용 말투 정책이 아직 시스템화되어 있지 않다

학생 대시보드와 학생 리포트 페이지에는 친근한 문구가 일부 존재하지만, 모든 학생 대상 페이지·상태 메시지·리포트 문장·오류 메시지·동기부여 문구에 일관된 말투 정책이 적용된 것은 아니다.

학생에게 제공되는 모든 정보는 **“다정하고 친근한 남자 선생님 말투”** 로 통일되어야 한다.

## 3. 개선 목표

### 3.1 핵심 목표

학생이 회원가입부터 로그인, 학생 정보 연결, 대시보드 진입, 리포트 조회, 학습 계획 확인, 배지 획득, 내 풀이 분석까지 자연스럽게 사용할 수 있도록 한다.

동시에 상용화 이후 교사를 고객으로 받더라도 Super Admin과 Teacher의 권한이 혼동되지 않도록 역할·라우팅·데이터 접근 범위를 분리한다.

### 3.2 세부 목표

| 목표 | 설명 |
|---|---|
| 학생 가입 가능 | 회원가입 화면에서 학생 유형을 선택할 수 있어야 한다. |
| 학생 로그인 라우팅 | 학생 로그인 성공 후 `/student`로 이동해야 한다. |
| 학생-프로필 연결 | 학생 계정이 기존 `students` 레코드와 안전하게 연결되어야 한다. |
| Super Admin 분리 | 서비스 운영자인 나는 전체 시스템을 관리하고, 고객 교사는 본인 범위만 관리해야 한다. |
| 교사 고객 확장성 | 개인 과외 교사뿐 아니라 학원·강사 조직까지 수용 가능한 권한 구조를 준비한다. |
| 학생 전용 UX 강화 | 학생에게는 친근하고 동기부여 중심의 화면과 문구를 제공한다. |
| 접근 권한 강화 | 각 사용자는 자기 역할과 연결된 데이터만 접근해야 한다. |
| 베타 운영 적합성 | 학생 가입/연결 오류가 발생해도 교사 또는 Super Admin이 쉽게 복구할 수 있어야 한다. |

## 4. 권장 사용자 역할 체계

### 4.1 최상위 역할

| 역할 | 권장 role 값 | 설명 |
|---|---|---|
| Super Admin | `super_admin` 또는 `admin` | 플랫폼 전체를 운영하는 나. 모든 테넌트·고객·시스템 설정 관리. |
| Teacher | `teacher` | 고객으로서의 교사. 본인 또는 본인 조직의 학생·학부모·리포트 관리. |
| Parent | `parent` | 자녀 학습 리포트와 학습 계획을 열람·지원하는 사용자. |
| Student | `student` | 본인 리포트·학습 계획·배지·자기 분석을 이용하는 사용자. |

### 4.2 교사 역할의 내부 확장 구조

상용화 이후를 고려하면 `teacher` 아래에 세부 권한을 둘 것을 제안한다.

| 세부 유형 | 설명 | 예시 권한 |
|---|---|---|
| `teacher_owner` | 개인 과외 교사 또는 학원 원장 | 본인 테넌트 결제, 학생/학부모/강사 초대, 전체 리포트 관리 |
| `teacher_instructor` | 학원 소속 강사 | 배정된 학생·반·수업·리포트만 관리 |
| `teacher_assistant` | 보조 강사 또는 운영 스태프 | 수업 기록·숙제 관리 등 제한 권한 |

MVP에서는 `teacher` 하나로 시작하되, 데이터 모델에는 향후 `teacher_role` 또는 `membership_role`을 추가할 수 있게 설계하는 것이 좋다.

### 4.3 테넌트/조직 구조 제안

교사가 고객이 되는 구조에서는 아래 개념이 필요하다.

| 개념 | 목적 |
|---|---|
| `tenant` 또는 `organization` | 한 명의 개인 교사 또는 하나의 학원/조직을 나타냄 |
| `tenant_id` | 학생, 학부모, 리포트, 수업, 결제, 설정을 고객 단위로 격리 |
| `memberships` | 한 테넌트 안에서 교사·강사·스태프 권한을 관리 |
| `billing_account` | 고객 교사의 구독·결제 상태 관리 |
| `feature_flags_by_tenant` | 특정 고객에게만 기능 베타 제공 |

| 학생 동기부여/목표 관리 | 정책/템플릿 관리 | 본인 학생 목표 설정 | 자녀 목표 지원 | 본인 목표 수행 |
| 개인정보 삭제/복구 | 법적·운영 권한 | 본인 테넌트 요청/일부 실행 | 요청 가능 | 요청 가능 |
| 고객 지원 impersonation | 감사 로그 하에 가능 | ❌ | ❌ | ❌ |

### 5.2 데이터 접근 원칙

1. **Super Admin**은 플랫폼 운영을 위해 전체 데이터를 볼 수 있지만, 고객 신뢰를 위해 접근 로그와 사유 기록이 필요하다.
2. **Teacher**는 자신이 소유하거나 소속된 테넌트의 데이터만 접근해야 한다.
3. **Parent**는 연결된 자녀 데이터만 접근해야 한다.
4. **Student**는 본인 데이터만 접근해야 한다.
5. 학생은 미성년자일 가능성이 높으므로, 학생 직접 가입 시 교사 또는 학부모 승인/연결 절차가 필요하다.

## 6. 사용자별 최초 가입부터 대시보드까지의 흐름

### 6.1 Super Admin / 나

#### 가입 및 계정 생성

- 공개 회원가입에서 생성하지 않는다.
- Supabase 또는 내부 관리 절차로 최초 1개 Super Admin 계정을 생성한다.
- `users.role = 'super_admin'` 또는 `admin`으로 지정한다.
- 2FA 또는 최소한 강력한 비밀번호 정책을 적용한다.

#### 로그인 후 이동

- `/super-admin` 또는 `/system-admin`으로 이동한다.
- 현재 `/admin`은 교사용 대시보드로 쓰이고 있으므로, 장기적으로 Super Admin과 혼동되지 않도록 분리하는 것을 권장한다.

#### 대시보드 구성

| 영역 | 내용 |
|---|---|
| 전체 운영 현황 | 활성 테넌트 수, 가입 교사 수, 학생 수, 학부모 수, 월간 리포트 수 |
| 비용 모니터링 | Gemini 호출 수, 모델별 토큰/비용, PDF 생성 수, 이미지 업로드량 |
| 고객 관리 | 교사/학원 계정 승인, 정지, 플랜 변경, 지원 티켓 |
| Feature Flags | 기능별 활성화, 테넌트별 베타 오픈, 롤백 스위치 |
| 품질 관리 | AI 분석 실패율, 파싱 오류, Sentry 에러, RLS 이상 탐지 |
| 데이터 관리 | 백업 상태, 개인정보 삭제 요청, 로그 보존 정책 |
| 공지/운영 | 전체 공지, 교사 대상 업데이트 안내, 장애 공지 |

#### Super Admin unique 기능 제안

- 테넌트별 사용량/비용 대시보드
- AI 모델 라우팅 정책 관리
- 기능 플래그/베타 그룹 관리
- 교사 고객 온보딩 상태 관리
- 고객별 데이터 접근 감사 로그
- 전체 배지/목표/리포트 템플릿 정책 관리
- 고객 지원용 “읽기 전용 impersonation” 기능
- 테넌트별 위험 알림: 과도한 AI 호출, 실패율 증가, RLS 오류 등

### 6.2 Teacher / 고객 교사

#### 가입 및 계정 생성

상용화 후 교사는 고객이므로 두 가지 가입 흐름이 필요하다.

| 방식 | 설명 | 권장 단계 |
|---|---|---|
| 초대/승인형 가입 | Super Admin이 교사 계정을 승인하거나 초대 | 베타/MVP |
| 셀프 가입 + 결제/승인 | 교사가 직접 가입 후 무료 체험 또는 결제 | 정식 상용화 |

MVP에서는 무분별한 AI 비용 발생을 막기 위해 교사 가입 후 즉시 전체 기능을 열기보다, `pending`, `trial`, `active`, `suspended` 같은 계정 상태를 둘 것을 권장한다.

#### 로그인 후 이동

- 현재 `/admin`이 교사 대시보드 역할을 한다.
- 장기적으로는 `/teacher`로 변경하거나, UI 명칭을 “교사 대시보드”로 명확히 해야 한다.

#### 대시보드 구성

| 영역 | 내용 |
|---|---|
| 오늘 수업 | 오늘 수업 학생, 최근 취약점, 숙제 상태, 최근 리포트 |
| 학생 관리 | 학생 등록, 학년/학교/학습 스타일, 학생 계정 연결 |
| 학부모 관리 | 학부모 계정 생성, 자녀 연결, 알림 발송 |
| 리포트 생성 | 레벨/시험/주간/월간/통합/반기/연간 리포트 생성 |
| 수업 관리 | 수업 일정, 수업 기록, 숙제, 진도 관리 |
| 성장 분석 | 학생별 Growth Loop, 메타프로필, 취약점 해결 상태 |
| 전략 관리 | AI 처방 실행 상태, 효과 추적, 다음 전략 조정 |
| 고객 계정 정보 | 현재 플랜, 사용량, 남은 분석 횟수, 결제 상태 |

#### Teacher unique 기능 제안

- 학생/학부모 초대 링크 발급
- 학생 계정 연결 코드 생성
- 리포트 템플릿/브랜딩 설정
- 교재·커리큘럼 템플릿 관리
- 학부모에게 보낼 요약 메시지 자동 생성
- 수업 전 “오늘 수업 브리핑” 자동 생성
- 월말 학부모 상담 자료 자동 생성
- 교사용 매출/운영 보조 기능: 결석/보강/수업료 메모, 단 MVP 이후 권장
- 학원 확장용 반/그룹 관리, 강사별 담당 학생 배정

### 6.3 Parent / 학부모

#### 가입 및 계정 생성

학부모는 원칙적으로 교사 또는 Super Admin의 초대/연결을 통해 가입해야 한다.

권장 흐름:

1. 교사가 학생을 등록한다.
2. 교사가 학부모 이메일을 입력하고 초대 또는 계정 생성을 진행한다.
3. 학부모가 가입/로그인한다.
4. `students.parent_id`로 자녀와 연결된다.
5. 학부모는 연결된 자녀의 리포트만 조회한다.

#### 로그인 후 이동

- `/parent`

#### 대시보드 구성

| 영역 | 내용 |
|---|---|
| 자녀 선택 | 여러 자녀가 있을 경우 자녀별 전환 |
| Growth Loop 현황 | Baseline, 시험, 주간, 월간, 반기, 연간 진행 상태 |
| 최근 리포트 | 최근 리포트 목록, 핵심 점수/성장률/습관 지표 |
| 성장 그래프 | 성적 추이, 수학 역량 레이더, 학습 습관 추이 |
| 이번 주 지원 | 가정에서 도와줄 체크리스트, 숙제/복습 안내 |
| 알림 | 새 리포트 생성, 주간 요약, 상담 필요 알림 |
| 질문/피드백 | 교사에게 간단 질문 또는 상담 요청 |

#### Parent unique 기능 제안

- “이번 주 우리 아이에게 해줄 한마디” 자동 생성
- 가정 학습 지원 체크리스트
- 리포트별 “학부모 요약 3줄”
- 자녀 성장 축하 카드
- 상담 요청/질문 기능
- 자녀의 배지/목표 달성 알림
- 학생에게 보여줄 수 있는 “칭찬 메시지 공유” 기능

### 6.4 Student / 학생

#### 가입 및 계정 생성

학생 가입은 반드시 가능해야 한다. 다만 미성년자 보호와 데이터 연결 정확성을 위해 다음 중 하나의 방식으로 진행하는 것이 좋다.

| 방식 | 설명 | 권장도 |
|---|---|---|
| 교사가 학생 계정 생성 | 교사가 학생 이메일/초기 비밀번호를 생성하고 `students.user_id` 연결 | 베타/MVP 추천 |
| 학생 직접 가입 + 연결 코드 | 학생이 회원가입에서 “학생” 선택 후 교사가 발급한 코드로 본인 학생 레코드 연결 | 정식 서비스 추천 |
| 학부모 승인형 학생 가입 | 학생 가입 후 학부모 또는 교사가 승인해야 활성화 | 미성년자 보호에 적합 |

학생이 직접 가입할 때는 가입 화면에 다음 입력 또는 후속 단계가 필요하다.

- 이름
- 이메일
- 비밀번호
- 사용자 유형: 학생
- 초대 코드 또는 학생 연결 코드
- 담당 선생님/학원 선택 또는 코드 기반 자동 연결
- 개인정보/보호자 동의 안내

#### 로그인 후 이동

- `/student`

#### 대시보드 구성

| 영역 | 내용 |
|---|---|
| 오늘의 응원 메시지 | 학생의 최근 상태에 맞춘 짧고 따뜻한 동기부여 |
| 나의 성장 요약 | 최근 점수, 학습 진도, 총 리포트, 획득 배지 |
| 내 학습 계획 | 오늘/이번 주 해야 할 일, 완료 체크 |
| 내 리포트 | 레벨/시험/주간/월간/반기/연간/내 풀이 분석 조회 |
| 나의 배지 | 획득 배지, 다음 배지까지 남은 조건, 포인트 |
| 내 풀이 분석 | 문제풀이 사진 업로드, 누적 데이터와 비교 분석 |
| 성장 예측 | 1개월/3개월/6개월 목표, 달성 가능성 |
| 선생님 한마디 | 리포트 또는 수업 기록 기반 피드백 |

#### Student unique 기능 제안

- 배지/포인트/레벨 시스템
- “다음 배지까지 2문제 남았어!” 같은 목표 안내
- 오늘의 미션 카드
- 학습 스트릭
- 학생용 오답 회복 여정
- 내 풀이 분석 히스토리
- 스스로 목표 설정하기
- 선생님에게 질문하기
- 나만의 성장 앨범: 처음 리포트부터 현재까지 성장 스토리

## 7. 학생 가입 기능 개선 요청

### 7.1 회원가입 UI 수정 요청

회원가입 화면의 사용자 유형에 `학생`을 추가해야 한다.

현재 상태:

- `선생님`
- `학부모`

개선 후:

- `선생님`
- `학부모`
- `학생`

추가로 Super Admin은 공개 회원가입 선택지에 노출하지 않는다. Super Admin 계정은 내부 운영 절차로만 생성한다.

### 7.2 학생 선택 시 안내 문구

학생을 선택하면 아래 안내 문구를 제공한다.

> 학생 계정은 담당 선생님이 등록한 학생 정보와 연결되어야 사용할 수 있어요. 선생님에게 받은 초대 코드가 있다면 입력해 주세요. 코드가 없다면 가입 후 선생님에게 연결을 요청할 수 있어요.

### 7.3 학생 연결 방식

학생 계정과 학생 레코드는 다음 방식 중 하나로 연결한다.

#### MVP 권장 방식: 교사 생성/연결형

1. 교사가 학생 관리 화면에서 학생을 등록한다.
2. 교사가 학생 계정 생성 버튼을 누른다.
3. 시스템이 Auth user와 `users.role = student`를 생성한다.
4. `students.user_id = auth.users.id`로 연결한다.
5. 학생은 로그인 후 `/student`로 이동한다.

#### 정식 서비스 권장 방식: 학생 직접 가입 + 연결 코드

1. 교사가 학생별 연결 코드를 생성한다.
2. 학생이 회원가입에서 `학생`을 선택한다.
3. 학생이 연결 코드를 입력한다.
4. 시스템이 코드의 유효성을 확인한다.
5. 기존 `students` 레코드와 학생 계정을 연결한다.
6. 학생은 `/student`로 이동한다.

### 7.4 학생 연결 실패 시 UX

학생 정보 연결에 실패하면 다음처럼 안내한다.

> 아직 선생님이 등록한 학생 정보와 연결되지 않았어요. 괜찮아요. 담당 선생님께 “학생 계정 연결”을 요청하면 바로 사용할 수 있어요.

버튼:

- “연결 코드 다시 입력하기”
- “선생님께 문의하기”
- “로그아웃”

## 8. 학생 페이지 말투 정책

### 8.1 원칙

학생이 보는 리포트 및 학생이 접근 가능한 모든 페이지는 **다정하고 친근한 남자 선생님 말투**로 정보를 제공해야 한다.

말투 원칙:

1. 평가보다 성장 중심으로 말한다.
2. 틀림을 비난하지 않는다.
3. “할 수 있다”는 느낌을 준다.
4. 학생에게 직접 말하듯 설명한다.
5. 너무 유아적이지 않게, 그러나 부담스럽지 않게 말한다.
6. 어려운 교육 용어는 쉽게 풀어준다.
7. 결과보다 다음 행동을 명확히 알려준다.

### 8.2 문체 예시

| 상황 | 지양 | 권장 |
|---|---|---|
| 오답 설명 | “개념 이해가 부족합니다.” | “이 부분은 아직 조금 헷갈릴 수 있어. 선생님이랑 한 번만 더 정리하면 금방 좋아질 거야.” |
| 점수 하락 | “성적이 하락했습니다.” | “이번에는 조금 아쉬웠지만 괜찮아. 어디서 흔들렸는지 찾았으니까 다음엔 더 잘할 수 있어.” |
| 약점 안내 | “문제 해석 능력이 낮습니다.” | “문제를 읽을 때 중요한 조건을 하나 놓친 것 같아. 앞으로는 밑줄을 치면서 읽어보자.” |
| 개선 전략 | “오답노트를 작성하세요.” | “틀린 문제는 그냥 넘기지 말고, ‘왜 틀렸는지’ 한 줄만 적어보자. 그게 실력을 확 올려줄 거야.” |
| 배지 획득 | “배지를 획득했습니다.” | “와, 이번 배지 얻었네! 꾸준히 해낸 게 진짜 멋지다.” |
| 목표 미달 | “목표를 달성하지 못했습니다.” | “이번 목표는 아직 다 못 끝냈지만 괜찮아. 남은 부분부터 같이 차근차근 해보자.” |

### 8.3 학생 리포트 문장 구조

학생 리포트는 아래 순서로 구성하는 것을 권장한다.

1. **칭찬/인정**: “이번에 이런 점이 좋았어.”
2. **현재 상태 설명**: “지금은 이 부분을 연습하면 좋아.”
3. **원인 설명**: “왜냐하면 이런 문제에서 조건을 놓쳤기 때문이야.”
4. **구체적 행동**: “오늘은 10분만 이 유형 3문제를 풀어보자.”
5. **응원**: “충분히 할 수 있어. 선생님이 보기엔 방향이 좋아.”

### 8.4 학생용 금지 표현

학생 화면에서는 아래 표현을 피한다.

- “부족하다”
- “낮다”
- “실패”
- “위험”
- “문제 있음”
- “반드시 해야 함”
- “심각”
- “미달”

필요한 경우 다음처럼 바꾼다.

| 기존 | 변경 |
|---|---|
| 부족 | 더 연습하면 좋아질 부분 |
| 낮음 | 아직 자라는 중 |
| 실패 | 이번엔 목표까지 조금 남음 |
| 위험 | 지금 챙기면 좋아지는 신호 |
| 심각 | 먼저 살펴보면 좋은 부분 |

| 반/그룹 관리 | 학원·그룹 수업 확장 대응 |
| 강사별 담당 학생 배정 | 학원형 고객 확장 대응 |

### 9.3 Parent 추가 제안

## 10. 접근 권한 및 RLS 개선 요청

### 10.1 Super Admin

- 모든 테넌트와 사용자를 관리할 수 있어야 한다.
- 단, 개인정보 접근 시 감사 로그를 남겨야 한다.
- 지원 목적의 데이터 접근은 사유와 시간, 접근한 사용자 ID를 기록한다.

### 10.2 Teacher

- 본인 테넌트의 학생, 학부모, 리포트, 수업, 숙제만 접근해야 한다.
- 다른 교사/학원 고객의 데이터에는 접근할 수 없어야 한다.
- AI 생성 기능은 플랜과 사용량 제한을 따라야 한다.

### 10.3 Parent

- `students.parent_id = auth.uid()` 또는 연결 테이블 기반으로 자녀 데이터만 접근한다.
- 리포트 생성/수정/삭제는 불가하다.
- 자녀의 학습 계획 체크 또는 피드백 기능은 제한적으로 허용할 수 있다.

### 10.4 Student

- `students.user_id = auth.uid()` 기준으로 본인 데이터만 접근한다.
- 본인 리포트, 본인 학습 계획, 본인 배지, 본인 내 풀이 분석만 접근한다.
- 다른 학생, 다른 학부모, 다른 교사의 데이터에는 접근할 수 없어야 한다.

## 12. 수용 기준

### 12.1 학생 가입 수용 기준

- 회원가입 화면에서 학생을 선택할 수 있다.
- 학생 회원가입 시 `users.role = student`가 생성된다.
- 학생 계정이 `students.user_id`와 연결된다.
- 학생 로그인 후 `/student`로 이동한다.
- 연결되지 않은 학생은 친절한 안내 화면을 본다.
- 학생은 본인 리포트만 볼 수 있다.
- 학생은 본인 배지와 학습 계획만 볼 수 있다.

### 12.2 Super Admin/Teacher 분리 수용 기준

- Super Admin은 공개 회원가입으로 생성되지 않는다.
- Super Admin은 전체 고객/테넌트/시스템 설정에 접근할 수 있다.
- Teacher는 전체 시스템이 아니라 본인 테넌트 데이터만 접근한다.
- Teacher는 다른 교사의 학생·학부모·리포트에 접근할 수 없다.
- Teacher 고객의 AI 사용량과 플랜 상태가 추적된다.
- `/admin` 또는 `/teacher`의 역할 명칭이 혼동되지 않는다.

### 12.3 학생용 말투 수용 기준

- 학생 대시보드, 학생 리포트, 내 풀이 분석, 배지, 학습 계획, 오류 메시지는 모두 학생 친화 말투를 사용한다.
- 학생에게 부정적 낙인을 줄 수 있는 표현은 사용하지 않는다.
- 모든 리포트는 “현재 상태 → 다음 행동 → 응원” 구조를 포함한다.
- 배지/성장/목표 미달 상황에서 각각 적절한 격려 메시지를 제공한다.

## 13. QA 시나리오

### 13.1 학생 가입/로그인

1. 학생이 회원가입 화면에서 `학생`을 선택할 수 있다.
2. 학생이 초대 코드 없이 가입하면 연결 대기 화면을 본다.
3. 학생이 올바른 연결 코드를 입력하면 기존 학생 레코드와 연결된다.
4. 학생 로그인 후 `/student`로 이동한다.
5. 학생은 본인 리포트 상세 페이지를 볼 수 있다.
6. 학생은 다른 학생의 리포트 URL로 직접 접근해도 차단된다.

### 13.2 교사 고객 권한

1. 교사는 본인 학생 목록만 볼 수 있다.
2. 교사는 본인 학생의 학부모만 연결할 수 있다.
3. 교사는 다른 교사의 학생 ID로 리포트 생성 요청을 해도 거부된다.
4. 교사는 Super Admin 기능 메뉴를 볼 수 없다.

### 13.3 Super Admin 권한

1. Super Admin은 모든 테넌트 목록을 볼 수 있다.
2. Super Admin은 특정 교사 계정을 정지/복구할 수 있다.
3. Super Admin이 고객 데이터에 접근하면 감사 로그가 남는다.
4. Super Admin은 Feature Flag를 테넌트 단위로 조정할 수 있다.

### 13.4 학생 말투 검수

1. 학생 리포트에 “부족”, “실패”, “위험” 등 부정 표현이 직접 노출되지 않는다.
2. 오답 설명은 친근한 선생님 말투로 제공된다.
3. 목표 미달 상황에서도 다음 행동과 응원 메시지가 함께 제공된다.
4. PDF로 저장된 학생 리포트에도 동일한 말투가 유지된다.

## 14. 권장 라우트 구조

현재 `/admin`이 교사용 대시보드로 사용되고 있으므로 상용화를 고려하면 아래 중 하나를 선택해야 한다.

### 안 A: 장기적으로 명확한 분리

| 역할 | 라우트 |
|---|---|
| Super Admin | `/super-admin` |
| Teacher | `/teacher` |
| Parent | `/parent` |
| Student | `/student` |

장점: 역할 혼동이 없다. 상용화에 적합하다.

### 안 B: 현재 구조 유지 + 명칭 보정

| 역할 | 라우트 |
|---|---|
| Super Admin | `/system` 또는 `/super-admin` |
| Teacher | `/admin` |
| Parent | `/parent` |
| Student | `/student` |

장점: 현재 코드 변경량이 적다. 단, UI에서는 `/admin`을 “교사 대시보드”라고 명확히 표시해야 한다.

## 15. 결론

이번 개선은 단순히 회원가입 화면에 “학생” 옵션을 하나 추가하는 작업이 아니다.

학생 기능이 이미 존재하는 만큼, 학생이 실제로 가입하고 본인 데이터와 연결되어 리포트·배지·학습 계획·내 풀이 분석을 사용할 수 있는 완전한 흐름을 만들어야 한다.

동시에 상용화 이후 교사가 고객이 되는 구조에서는 Super Admin과 Teacher의 권한을 명확히 분리해야 한다. Super Admin은 플랫폼 전체를 운영하는 역할이고, Teacher는 본인 학생과 학부모를 관리하는 고객이다.

따라서 이번 개선의 핵심은 다음 3가지다.

1. **학생 가입과 학생 계정 연결을 완성한다.**
2. **Super Admin / Teacher / Parent / Student의 권한과 서비스를 명확히 분리한다.**
3. **학생이 보는 모든 화면은 다정하고 친근한 남자 선생님 말투로 통일한다.**

이 세 가지가 해결되어야 베타 서비스가 실제 사용자 흐름에서 자연스럽게 작동하고, 향후 개인 과외 교사뿐 아니라 학원·강사 조직까지 확장 가능한 상용 서비스로 발전할 수 있다.
