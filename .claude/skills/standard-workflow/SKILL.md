# 표준 워크플로우 (Loop Engineering)

원칙은 루트 [`../../../CLAUDE.md`](../../../CLAUDE.md), 하네스 메커니즘은 [`../../HARNESS.md`](../../HARNESS.md) 참조.
핵심: **만드는 주체(Maker)와 평가하는 주체(Critic)를 분리**하고, **승인 강도를 위험에 비례**시킨다.

## 작업 루프 (0→12)

```
0.  컨텍스트 로드 — MEMORY.md 확인, code-index MCP로 관련 코드 인덱싱        [컨텍스트 설계]
1.  현 상태 동기화 — git 상태/원격/untracked. 머지됐으면 main + pull --ff-only
2.  다음 작업 선정 — 리스크 최소화 우선. 자율영역 ↔ 인간필요영역으로 분해     [분해]
3.  선정 이유 — 사용자가치/리스크/데이터신뢰성/일정/UX 기준
4.  계획 + 완료정의(DoD) — 범위/비범위/변경파일/검증 + "끝났다"의 체크리스트  [완료정의]
5.  위험등급별 승인 게이트 — 아래 표 참조                                     [감각]
6.  새 작업 브랜치 — 최신 main 기준, 의도가 드러나는 이름(오래된 브랜치 재사용 금지)
7.  구현(Maker) — 승인 계획 내에서만. 범위·위험 커지면 멈추고 재보고
8.  Critic 독립 평가 게이트 — critic 서브에이전트가 채점→임계(7/10) 미달 시 보정 루프.
       만든 주체가 스스로 합격 판정 금지.                                     [완료정의·관찰가능성]
9.  구현 후 검증 — 변경 부위 타깃 테스트 우선. 결과를 PASS/MINOR/BLOCKER + "왜 안전한가"
10. 커밋·푸시·PR — 의도한 파일만 stage. PR 본문: Summary/Verification/Notes
11. PR 리뷰 대응 — 봇 맹신 금지. "타당한가" ≠ "지금 반영 무해한가" 분리. 좁게 수정·재검증·resolve
12. 머지 루틴 → MEMORY.md 갱신(무엇을/왜/다음) → 0번으로 회귀                  [메모리]
```

## 위험등급별 승인 (5단계 게이트)

일률 승인은 비실용적이다. **고위험 20%에만 인간 승인을 집중**한다.

| 등급 | 대상 | 게이트 |
|------|------|--------|
| 🟢 저위험 | 문서, `.claude`, 설정, 테스트 *추가* | 계획 명시 후 **자율 진행** + Critic 게이트로 대체 |
| 🟡 중위험 | 일반 UI/로직, 새 컴포넌트 | 계획 제시 → **짧은 확인**(이의 없으면 진행) |
| 🔴 고위험 | DB/마이그레이션, 인증/권한, 개인정보·학생데이터, 결제, **AI 판단 결과**, 핵심 진입경로 | 구현 전 **명시 승인 필수** + 롤백 전략 |

## 실패 복구 (같은 프롬프트 재시도 금지)

실패를 3분류하고 유형별로 다른 처방을 쓴다:
- **컨텍스트 부족** → 관련 파일/스키마/예시를 더 준다(code-index 활용).
- **방향 오류** → spec/DoD를 다시 합의한다. 구현이 아니라 계획으로 돌아간다.
- **구조적 충돌** → 접근 자체를 바꾼다. 작게 쪼개 격리 테스트한다.

## 역할 분담 (서브에이전트)

| 역할 | 담당 | 일 |
|------|------|----|
| 고추론 | 메인 세션 + `critic` | 작업선정·계획·위험판단·리뷰 타당성·검증해석·DB/보안/데이터신뢰성 판단 |
| Maker | 메인 세션 | 승인된 계획대로 구현·문서·테스트·커밋/푸시/PR |
| 점검 | 단순 서브에이전트 | git 상태, 로그/테스트 수집, PR 체크 상태, 반복 smoke |

## 항상 묻는 핵심 질문 (감각)

- 사용자가 실제로 얻는 가치가 좋아졌는가?
- 데이터 신뢰성이 높아졌는가?
- 실패했을 때 성공처럼 보이지 않는가?
- 사용자가 다음 행동을 알 수 있는가?

## PR 규율

- **1 PR = 1 의도.** 문서/UI/DB/테스트/설정을 섞지 않는다. 베타 직전엔 작고 되돌리기 쉬운 변경만.
- 분할 순서: DB(마이그레이션) → 백엔드/API → UI → 테스트 → 문서. 기준은 [`../report-harness/SKILL.md`](../report-harness/SKILL.md).
- 커밋엔 의도한 파일만. `scratch/`·임시·로컬설정·테스트 산출물·무관한 untracked는 제외.

## DB는 별도 등급

마이그레이션은 idempotency 확인. 기존 테이블/인덱스/정책·migration history 불일치를 함부로 고치지 않는다.
베타/운영 직전엔 DB 구조 변경을 기본 회피. 변경 시 계획·검증·롤백 전략 필수.
